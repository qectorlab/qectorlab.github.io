Do not overclaim.
Do not present QECTOR as a universal speed leader, production SaaS, OEM-ready embedded decoder, or real-time hardware QEC infrastructure unless a specific reproducible artifact proves that exact statement.

Correct baseline comparison

Use PyMatching, Stim, and Sinter as reference tools. QECTOR is positioned for packaged multi-decoder experiments, reproducibility, BP-OSD/LDPC workflows, GPU batch evaluation, workbench exports, and commercial licensing clarity.

Honest limitations

Known limitations from the README.

Union-Find accuracy

Union-Find is roughly 3× less accurate than MWPM across measured d=3–9 repetition and surface-code workloads. Use Blossom or SparseBlossom when accuracy matters.

Threshold curves

Single-round code-capacity noise does not produce surface-code threshold curves. For genuine threshold curves, use circuit-level noise via Stim DEM with BeliefMatching or qector_sinter_decoders().

Degenerate cases

SparseBlossom batch may return different but valid corrections than single-shot on degenerate cases. This is benign matching degeneracy, not an error.

GPU threshold

GPU wins only at sufficient batch sizes, typically 4096 or more. Measure on your own hardware before quoting speedup.

CUDA availability

CUDABatchDecoder raises a clean RuntimeError when no driver is present. Check availability before construction.

Artifact scope

Every public performance claim should identify package version, OS, CPU/GPU, code, distance, shots, seed, and dependency versions.

Reproduce first

Benchmark evidence is scoped. Re-run local tests before quoting technical, commercial, or hardware-readiness claims.
