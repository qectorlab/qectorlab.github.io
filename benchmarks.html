Benchmarks

Reproducible QEC Decoder Evidence

All benchmark results are produced by committed scripts, stamped with git commit and environment versions, and hashed for artifact integrity. No fabricated figures. Limitations are stated explicitly and re-checked on every release.

Methodology

How QECTOR benchmarks are produced

Environment capture

Every benchmark artifact records: CPU model (AMD64 Family 23 Model 96), Python version (3.11.0), Stim version (1.16.0), PyMatching version (2.4.0), QECTOR version, and the exact git commit hash. Figures in this document trace to that build.

Shot counts and intervals

LER comparisons use Wilson 95% confidence intervals throughout. Competitive head-to-head runs use 40,000 shots per point. Belief-matching runs use 8,000 shots per point (higher cost per shot). All intervals are reported — overlapping CIs confirm parity.

Latency measurement

Hot-path benchmarks use 5,000 trials with 500 warmup calls. p50 / p90 / p95 / p99 percentiles are reported. Cold-path timing (including Python dispatch overhead) is separated from hot-path (steady-state decode only). Rust-native nanosecond resolution via BenchmarkSuite.

Artifact hashing

The due-diligence bundle (python scripts/run_due_diligence_bundle.py) hashes all JSON and CSV artifacts with SHA-256 and stamps the git commit. Artifact-level reproducibility means external parties can verify results match the published build.

Reproduction commands

# Competitive LER sweep
python scripts/competitive_stim_ler.py \
  --distances 3 5 7 9 11 \
  --noise 0.005 \
  --shots 40000

# Belief-matching head-to-head
python scripts/competitive_belief_matching.py \
  --distances 3 5 7 \
  --noise 0.005 \
  --shots 8000

# Full reproducible benchmark
python scripts/run_competitive_benchmark.py \
  --code rotated_surface \
  --distances 3 5 7 9 \
  --decoders union_find blossom sparse_blossom \
  --trials 5000 --warmup 500 \
  --seed 1 \
  --out benchmark_results/competitive

# Due-diligence evidence bundle
python scripts/run_due_diligence_bundle.py
Validated scope: Claims are locked by the test suite. test_pymatching_parity_*, test_blossom_d15_no_gap, test_blossom_adaptive_k_regression and 411 additional tests. Any regression fails CI.

Belief-matching is an accuracy mode — it rebuilds the weighted matching per shot and is slower than plain MWPM. Use when logical error rate is the priority, not when latency is.

GPU Batch Decoding

CPU/GPU bit-identical batch throughput

GPU decoding results are bit-identical to CPU across all distances and batch sizes. Benchmarks run on GTX 1660 Ti. CUDA and OpenCL backends include transparent CPU fallback on runtime error and auto-recovery after 3 consecutive failures.

BP-OSD / qLDPC

Known Limitations

What these benchmarks do not claim

Latency vs PyMatching

QECTOR-Blossom is not the latency leader for single-shot exact MWPM. The gap is ~1.4× at d=3, ~3–4× at d=5–7, and ~5–6× at d=9–11. QECTOR's per-shot latency improves with DEM collapsing but PyMatching's optimised C++ remains faster for plain MWPM workloads.

Machine-specific numbers

All latency figures are from QECTOR's reference machine (AMD64 Family 23 Model 96, GTX 1660 Ti). Your hardware will produce different absolute numbers. The relationships (GPU vs CPU crossover, decoder ranking) are expected to hold qualitatively.

Single-machine validation

Before serious commercial outreach, QECTOR will add second-machine and Docker/Linux reproduction. These benchmarks are from a single validated environment — independent reproduction is planned for the next release milestone.

SparseBlossomDecoder

SparseBlossomDecoder is region-growing, not exact MWPM. ≥99% of small-code syndromes are minimum-weight, with weight gap ≤ 1 vs brute force. Use BlossomDecoder when exact minimum weight is required.
