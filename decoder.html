Decoder Selection

When to use which decoder

QECTOR ships multiple decoder backends. Choose based on your accuracy, latency, code family, and batch size requirements.

Exact MWPM

BlossomDecoder

Weight-optimal · Verified vs brute force
Use when you need minimum-weight matching or LER parity with PyMatching. Validated through d=15 for memory_x and memory_z.
Highest Accuracy

BeliefMatching

Up to 35.7% lower LER vs PyMatching (d=5)
Accuracy mode — recovers X–Z correlations via sum-product BP then exact MWPM. Higher per-shot cost; use when LER is the priority.
qLDPC / LDPC

BpOsdDecoder

BP + ordered statistics · [[72,12]] BB code
For non-graphlike CSS codes that matching cannot decode — bivariate-bicycle, hypergraph-product, bicycle. Within ~10% LER of reference ldpc package.
Near-Optimal

SparseBlossomDecoder

Region-growing · ≥99% optimal at small distances
Lower latency than Blossom with ≤1 weight gap vs brute force. Not exact MWPM — use BlossomDecoder when exact minimum weight is required.
Lowest Latency

UnionFindDecoder

1.4 µs p50 — real-time path
Fastest path on surface/repetition/toric matching graphs. Approximate — falls back to Blossom for safety. Best for real-time QEC control.
GPU Batch

CUDABatchDecoder / OpenCLBatchDecoder

Bit-identical to CPU · Transparent fallback
Large-batch offline Monte-Carlo sampling. CUDA for NVIDIA production. OpenCL for cross-vendor (NVIDIA / AMD / Intel). Win at batch ≥ 1024.
Adaptive

AutoDecoder

Calibrates CPU↔GPU crossover on your hardware
Routes to the GPU only when it is actually faster, measured on your machine. GPU output is bit-identical to CPU path.
CPU Batch

BatchDecoder / CPUBatchDecoder

4.1M dec/s · SIMD · Rayon parallel
High-throughput CPU batch sampling. Optimal at batch sizes 64–4096. Same accuracy as chosen backend decoder.

Get started

Standard install

pip install qector-decoder-v3

With development tools

pip install qector-decoder-v3[dev]

With Stim / Sinter

pip install qector-decoder-v3[stim]

Full extras (all optional deps)

pip install qector-decoder-v3[all]

Quick start — UnionFind decoder

from qector_decoder_v3 import UnionFindDecoder
import numpy as np

checks = [[0, 1], [1, 2], [2, 3]]
decoder = UnionFindDecoder(checks)

syndrome = np.array([0, 1, 0], dtype=np.uint8)
correction = decoder.decode(syndrome)
print("Correction:", correction)

Streaming multi-round

from qector_decoder_v3 import StreamingDecoder

stream = StreamingDecoder(checks, history_size=10)
for syndrome in syndrome_stream:
    correction = stream.update(syndrome)
    apply_correction(correction)
stream.flush()

    CUDABatchDecoder, OpenCLBatchDecoder

Architecture

Rust core with Python ecosystem

Rust core

Union-Find with path compression + union by rank. Flat edge storage for cache efficiency. Corrected algorithm — only active clusters corrected. Zero-copy NumPy interop via PyO3. GIL released during every decode call.

Blossom / SparseBlossom

Exact weighted MWPM via Edmonds primal-dual. SparseBlossomDecoder uses region-growing with BFS-per-pair, RadixHeap ordering, and exact DP for n ≤ 20 with Edmonds fallback for n > 20.

GPU backends

CUDA: NVRTC native kernel compilation, reusable device workspace, CPU fallback after runtime error, auto-recovery. OpenCL: dual-kernel (global/local memory), cross-vendor NVIDIA/AMD/Intel, same resilience model.

Rayon parallelism

BatchDecoder distributes across all CPU cores via Rayon. CPUBatchDecoder adds SIMD-optimised SoA layout for small/medium batches. AutoDecoder calibrates the CPU↔GPU crossover and routes accordingly.

Stim DEM loader

Correct detector graph: mechanisms = columns, detectors = rows. Handles repeat, shift_detectors, and ^ decomposition. Works without Stim installed. Emits matching weights, priors, observables.

Provenance

Every benchmark artifact stamps git commit, Python version, Stim/PyMatching versions, CPU details, and seed. The due-diligence bundle hashes all artifacts with SHA-256 so benchmark figures trace to a specific build.

Honest Limitations

Known limitations, re-checked every release

Benchmark figures are from QECTOR's reference machine. Your numbers will differ based on hardware. Latency ratios are honest comparisons against PyMatching on the same machine.

Latency vs PyMatching

QECTOR-Blossom matches PyMatching accuracy but is slower per shot at circuit level — roughly on par at d ≤ 5, ~5–6× behind at d = 9–11, gap widens beyond d = 13. For minimum single-shot latency on plain MWPM, PyMatching remains faster.

Belief-matching throughput

Belief-matching rebuilds the weighted matching per shot — a few shots/sec at d=7. It is the accuracy mode, not the fast path. Use when LER reduction matters more than throughput.

SparseBlossomDecoder exactness

SparseBlossomDecoder is region-growing, not exact MWPM. ≥99% of small-code syndromes are minimum-weight with a weight gap ≤ 1 vs brute force. Use BlossomDecoder when exact minimum weight is required.

GPU batch threshold

GPU transfer overhead means CUDA/OpenCL win decisively only at large batch sizes. AutoDecoder.calibrate() finds the crossover on your machine and keeps small batches on CPU. Do not assume GPU is always faster.
